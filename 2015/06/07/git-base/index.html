关于版本控制系统

版本控制系统（Version Control System）记录一个或一系列文件随时间的推移变化。如果需要，可以随时将文件还原到之前的某一状态。并且，版本控制系统记录了文件的所有变更记录，可以方便地找出谁操作了文件、谁引入了问题（issue）等等。

本地版本控制系统的最大缺点就是无法与他人协同工作，因此集中式版本控制系统应运而生。 其中有著名的CVS、Subverion等。集中式版本控制系统的工作原理如下：

分布式版本控制系统（Distributed Version Control Systems）

想像一下，对于集中式版本控制系统而言，如果很不幸，服务器挂了，带来的结果可能是毁灭性的。再者，假如服务器当机了几个小时（这还挺常见的），这段时间内是无法对服务器进行任何操作的。

分布式版本控制系统也有服务器与客户端之分，所不同的是，每一个客户端都是仓库（Repository）的镜像。同时，每一个客户端都具有独立完整的版本控制能力。就算服务器挂了，任意一台客户端都有能力将所有文件还原回去。结构如下：

Git的设计目标

Git最初由Linux社区开发，用来维护Linux内核代码的。其一开始设计就考虑了以下问题：

  • Speed 高速
  • Simple design 设计简单
  • Strong support for non-linear development(thousands of parallel branches) 强力支持非线性开发，支持成千上万的平行分支
  • Fully distributed 完全分布式
  • Able to handle large projects like the Linux kernal efficiently(speed and data size) 能够处理像Linux内核这样的大量项目，要求高速和大数据量。

Git的特点

记录文件快照而非文件差异：

其他版本控制系统，像Subversion，每次记录的是实际上是新版本文件相较于上一版本的差异。其工作原理图如下：

而Git的原理并非如此，对于每一次的提交，Git实际上是对所有文件进行了一次快照。每一次的快照保存的是文件版本的引用。如果文件没有改变就指向原来的文件，否则指向新文件。工作原理如下：

几乎所有操作都是本地执行的：

由于每一个客户端都拥有完整的服务器文件镜像，所以绝大多数的操作只需要在本地就能执行。查询日志、对比文件差异这种操作都只要在本地就能执行。甚至在没有网络的情况下也可以先将文件提交到本地系统，等连上服务器再进行推送。

保持数据的完整性：

文件在保存到Git之前都要时行校验和（Checksum）计算，该结果将作为数据的唯一标识和索引。

Git使用的校验和算法为SHA-1，通过对文件的内容或目录结构计算出一个40位的十六进制字符串。Git数据库就是使用该值作索引的，而非文件名。

Git管理文件的三种状态：

在Git内部，一个文件可能有三种状态：已提交（Commited）、已修改（Modified）、已暂存（Staged）。
