文件的状态

对于Git而言，文件有三种状态：

  • 已修改（Modified）
  • 已提交（Committed）
  • 已暂存（Staged）

新建一个learning git.txt文件，使用git status查看时，Git会提示有未被追踪（Untracked）的文件。

当使用git add添加时，文件的状态就会变为已暂存（也称已索引）。

当调用git commit时，才会把已暂存的文件提交到git仓库中。

也就是说，在工作目录（Working Directory）下的文件就两种状态：已追踪和未追踪。而已追踪的文件又可分为已修改和未修改两种情况。工作目录下的文件生命同期如图：

版本控制的一些基本概念

以下的内容适用于其他的版本控制系统，并不是Git独有。

软件开发通常都是需要团队合作的，这有点类似于一个多线程程序。写多线程程序要经常注意的一点就是线程同步。因为不同线程对同一变量进行修改很容易出现各种莫名其妙的错误。通常的解决方法就是加锁。

同步代码块、同步方法的作用是使临界资源在同一时刻（大概这么理解）只能被一个线程修改，其他线程等待。

这种方式显然不能很好地进行线性工作的，一个文件被一个人独占那其他人只能干等。

想像一个情景，如果A只是在文件的开头部分做修改，而B只是在文件的末尾进行修改。那么其实他们编辑结果是可以“并行”的。这就引入了第一个概念——合并（Merge）。
